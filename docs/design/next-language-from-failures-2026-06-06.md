# 다음 언어 설계 — Vais 실패 사례에서 도출한 원칙

**작성일**: 2026-06-06
**목적**: 기존 Vais를 "실패 사례 데이터"로 삼아, 백지에서 AI-native 언어를 설계하기 위한 원칙.
**근거**: 모든 항목은 이번 세션에 `vaisc`/`rustc`로 **실측**한 버그·함정·벤치마크에서 도출. 추측은 명시.

## 0. 이 문서의 위치

- 이 문서는 "Vais를 고치는 법"이 아니라, **"Vais가 왜 이런 문제를 구조적으로 만들었는가 →
  새 언어는 그 함정을 설계로 원천 차단"**을 다룬다.
- 동반 문서: `ai-native-competitiveness-assessment-2026-06-06.md` (경쟁력/시장 평가).
- 핵심 전제(실측): AI(LLM)는 **충분한 예제 컨텍스트가 있으면 신규 언어도 정확히 쓴다**
  (세션 후반 10/10). 단 **cold-start와 모호한 문법에서 틀린다**(세션 초반 Vec 3회 실패).
  → 새 언어의 제1목표 = **"AI가 cold-start에서도 틀릴 수 없는" 설계**.

## 1. 실패 사례 → 근본 원인 → 설계 원칙

### 실패 1: 단일문자 키워드의 식별자 충돌
**증상**: `struct A` (A=async), `fn f32` (f32=타입), `C` (C=continue) 등이 키워드와 충돌해
`found Async, expected identifier` 같은 cryptic 에러. AI(나)가 세션 내내 반복해서 밟음.
**근본 원인**: 키워드 공간과 식별자 공간이 겹친다. 토큰 절약을 위해 1글자 키워드를 썼는데,
1글자 식별자(특히 흔한 `A`,`C`,`f32`)와 충돌.
**실측 비용**: 토큰 절약 < 충돌로 인한 재작업. 벤치마크상 Rust가 총 22% 더 짧음.
**→ 설계 원칙 P1**: **키워드와 식별자 공간을 절대 겹치지 않게.** 키워드는 예약어로 명확히 분리하거나,
문맥 자유 문법으로 "이 위치엔 키워드만/식별자만"이 결정되게. **토큰 절약을 위해 모호성을 만들지 말 것.**
(실측 결론: "토큰 최소화"는 잘못된 목표. "AI 생성 정확도 최대화"가 옳은 목표.)

### 실패 2: 같은 기호의 문맥 의존 의미 (`~`)
**증상**: `x := ~a`에서 `~`가 bitwise-NOT이 아니라 **mut-shorthand**. AI가 Rust 직관으로 NOT으로 씀.
**근본 원인**: 한 기호가 위치에 따라 다른 의미. `~`는 `:=` 뒤에선 mut, expr에선 NOT.
**→ 설계 원칙 P2**: **한 기호 = 한 의미.** 같은 토큰이 문맥에 따라 다른 뜻을 갖지 않게.
mut은 `mut` 키워드로, NOT은 `not`/`bitnot`으로 — 명시적이고 유일하게.

### 실패 3: 표현 방식이 여러 개 (qualified vs unqualified)
**증상**: enum match에서 `Color::Red`는 에러, `Red`만 됨. AI가 Rust식 qualified로 씀.
**근본 원인**: "한 가지 쓰는 법"이 강제되지 않음. AI는 학습데이터(Rust)의 방식을 먼저 시도.
**→ 설계 원칙 P3**: **한 작업 = 정확히 한 가지 문법.** 선택지가 있으면 AI는 익숙한(=다른 언어의) 방식을
시도하고 틀린다. 가능하면 둘 다 허용해서 AI가 못 틀리게, 아니면 하나만 두되 에러가 정확히 안내.

### 실패 4: 빌더/생성자 문법 불일치 (turbofish)
**증상**: `Vec<i64>::new()`는 P001, `Vec::new()`만 됨. AI가 Rust turbofish로 씀.
**근본 원인**: 타입 인자 위치가 Rust와 다른데, 에러가 "고치는 법"을 안 줌(`found I64, expected expression`).
**→ 설계 원칙 P4**: **에러는 증상이 아니라 수정 코드를 줘야 한다.** (Rust의 `help: use ::<...>` gold standard.)
모든 흔한 실수에 "이렇게 고쳐라" + 정확한 코드. AI self-correction의 결정타.
(실측: Vais는 공통 에러는 우수하나 고유 함정엔 `help:` 없음 → AI 3회 실패.)

### 실패 5: 표준 라이브러리가 명시 import + 환경 의존
**증상**: `use std/vec` 빠뜨리면 C002. 그리고 프로젝트 루트 밖에선 `Cannot find std library`.
**근본 원인**: std가 toolchain-global이 아니라 경로/import 의존. Rust의 `vec![]`는 어디서나 동작.
**실측 비용**: AI가 import를 자주 빠뜨림. 환경 의존으로 "되는데 안 되는" 혼란.
**→ 설계 원칙 P5**: **핵심 std는 prelude로 항상 가용 + toolchain-global.** import 없이 `Vec`/`Map`/`sort`가
바로 되게. AI가 빠뜨릴 import 자체를 없앤다.

### 실패 6: 표현식 vs 문 경계 (`=> return`)
**증상**: match arm에 `=> return v` 안 됨, `=> { return v }` 필요. AI가 자연스럽게 전자로 씀.
**근본 원인**: arm은 표현식만, return은 문. 경계가 직관과 어긋남.
**→ 설계 원칙 P6**: **흔한 패턴은 추가 의식(ceremony) 없이.** `=> return`처럼 명백한 의도는 그대로 허용.
문법이 "AI가 자연스럽게 쓰는 것"을 거부하면 안 된다.

### 실패 7: 타입 폭 불일치가 codegen에서 silent (f32 전면)
**증상**: `x: f32 := 6.0`이 0.0/nan. f32 리터럴(double)이 f32 슬롯에 그냥 저장됨. 10개 site에서 동일.
**근본 원인**: 리터럴의 기본 타입(double)과 선언 타입(f32) 불일치를 **각 codegen site가 개별 처리** →
한 곳만 빠져도 silent miscompile. (narrow-int도 같은 클래스였음.)
**→ 설계 원칙 P7**: **타입 변환은 단일 지점(중앙 coercion)에서.** "literal → typed slot" 변환을 모든
store/arg/return/field/array가 거치는 한 곳으로. 산재하면 반드시 누락 생김. (이번에 8개 버그가 다 이 패턴.)
**→ 설계 원칙 P7b (검증)**: **컴파일 성공 ≠ 정답.** 값-정확성을 컴파일러 테스트의 1급 시민으로.
(실측: vaisdb 게이트가 "크래시 없음"만 봐서 정렬 오류를 통과시킴. VALUE CORRECTNESS 게이트로 보강.)

### 실패 8: 클로저 캡처가 call boundary에서 유실 (서비스 블로커)
**증상**: capturing closure를 sort_by에 넘기면 캡처가 사라져 **vaisdb 정렬이 무음으로 틀림**.
**근본 원인**: 클로저를 **env 없는 bare fn-ptr**로 lowering. 캡처를 prepend-param으로 넣었는데
fn(T,T)->i64 호출규약엔 env 슬롯이 없어, 경계 넘으면 캡처가 누락.
**→ 설계 원칙 P8**: **클로저는 처음부터 `{code, env}` 1급 값으로.** 모든 함수값이 env를 동반하는
단일 호출규약. "캡처 없는 건 bare fn-ptr, 있는 건 다른 표현" 같은 이중 표현이 이 버그의 근본.
**→ 메타 교훈**: 이 버그가 broad ABI 변경이 된 이유 = **호출규약을 나중에 안 바꿔도 되게 처음부터 통일**
안 해서. 새 언어는 함수값 표현을 day-1에 확정.

## 2. 공통 메타 패턴 (8개 실패를 관통하는 것)

| 메타 패턴 | 실패들 | 새 언어 원칙 |
|----------|--------|------------|
| **모호성** (한 토큰/위치 여러 의미) | 1,2,3,6 | 모호성 0. 한 토큰=한 의미, 한 작업=한 문법 |
| **산재한 변환/표현** (중앙화 안 됨) | 7,8 | 단일 지점 coercion + 단일 호출규약. day-1 확정 |
| **에러가 수정법을 안 줌** | 1,4 | `help:` + 수정 코드 (AI self-correction 핵심) |
| **컴파일≠정답 (silent miscompile)** | 7,8 | 값-정확성을 게이트 1급 시민 |
| **환경/import 의존** | 5 | prelude + toolchain-global |

**한 줄 요약**: Vais의 실패는 대부분 **"토큰 절약을 위해 모호성을 도입"** + **"변환/표현을 산재"**
+ **"컴파일만 검증"**에서 나왔다. 새 언어는 이 셋을 정확히 뒤집는다:
**모호성 0 + 중앙화 + 값-정확성 게이트.**

**+ 실측 입증된 추가 레버 (P9)**: 신규 언어의 "LLM 학습데이터 없음" 약점은 **검증된 예제 코퍼스**로
실용적으로 해소된다. cold-start AI 정확도가 예제 유무로 **1/5 → 5/5** (이 세션 실측, 최강 데이터).
사용자의 초기 직관("예제 잘 만들면 된다")이 데이터로 입증됨. 내 초기 회의("AI가 신규 언어 못 씀")는 정정.

## 3. 새 언어 설계 방향 (실패에서 도출)

### 3.1 표면 문법
- **영어스러운 키워드** (`fn`/`if`/`return`/`match` — AI 학습데이터 풍부). 단일문자 폐기.
- **키워드 ≠ 식별자 공간** (P1). 모호성 0 (P2,P3).
- **흔한 패턴은 ceremony 없이** (P6).
- **토큰 절약은 비목표.** 목표는 AI cold-start 정확도.

### 3.2 타입/의미
- **단일 지점 coercion** (P7) — literal↔typed 변환 한 곳.
- **클로저 = `{code, env}` 1급 값, 단일 호출규약** (P8) — day-1 확정.
- 명시성/결정론 (암시적 coercion 금지 — Vais가 이미 지향한 부분).

### 3.2b 예제 코퍼스 — 가장 강하게 입증된 레버 (P9)
**→ 설계 원칙 P9 (실측 입증, 이 세션 최강 데이터)**: **검증된 예제 코퍼스를 언어의 1급 인프라로.**
실험: 신규 AI(Vais 모름)에게 같은 5과제를 (A) 예제 없이 → **1/5**, (B) 관련 예제를 컨텍스트에 주고 →
**5/5**. 예제 코퍼스 하나로 cold-start 정확도 20%→100%.
- 문법 기능마다 **컴파일 검증된 예제**를 둔다 (Vais의 LIVING_SPEC가 이미 이 방향, 더 강화).
- AI가 cold-start에서 **관련 예제를 자동 참조**하는 구조 (검색/매칭이 별도 과제).
- AI 생성→`build` 검증→정답쌍 자동 축적으로 코퍼스를 키운다.
- **이것이 "신규 언어 = LLM 학습데이터 없음" 약점을 실용적으로 해소하는 핵심.**
  단 표본 5, 관련 예제일 때만 유효 — "어떤 예제를 언제" 매칭이 실전 과제.

### 3.3 도구/검증
- **에러에 `help:` + 수정 코드** (P4) — 모든 흔한 실수.
- **값-정확성 게이트 1급** (P7b) — 컴파일+런타임 정답 동시 검증.
- **prelude + toolchain-global std** (P5).

### 3.4 재활용 가능한 Vais 자산 (백지지만 0은 아님)
- ✅ LLVM codegen 파이프라인 (단, 단일 coercion 지점으로 재구성)
- ✅ 타입체커/추론 엔진
- ✅ self-hosting 노하우 + 빠른 프론트엔드 (check가 Rust 2배)
- ✅ 검증 인프라 (VALUE CORRECTNESS / sort-correctness 게이트 패턴)
- ❌ 단일문자 키워드, 토큰-절약 철학, 산재 coercion, bare-fn-ptr 클로저 (전부 폐기)

### 3.5 차별화 축 (실측이 가리킨 것)
- **컴파일 속도** — 프론트엔드는 이미 Rust 2배. LLVM 우회 디버그 백엔드 → AI 루프 압도 가능.
- **AI self-correction** — `help:` 자동수정 + 모호성 0 → AI가 1라운드에 수렴.
- **예제 코퍼스 자동 축적** — AI 생성→검증→정답쌍. cold-start 약점 보완 (사용자 가설, 실측 지지).
- ⚠️ **여전히 미해결: 생태계 해자.** 이건 언어 설계로 못 푼다 (경쟁력 보고서 §3 참조).

## 4. 다음 단계 (추측 아닌, 검증된 순서)

## 5. 정직한 한계 (잊지 말 것)

- 이 원칙들은 "AI가 더 잘 쓰는 언어"를 만든다 — 그러나 **"범용 시장 대체"는 별개**.
  생태계·채택 해자는 설계로 못 푼다 (경쟁력 보고서 결론).
- 표본이 작다 (버그 8 + 함정 7 + 벤치 5~10). 더 측정하면 원칙이 정제될 수 있다.
- "새로 만든다"는 큰 결정 — 이 문서는 **방향**이지 **시작 신호**가 아니다.
  §4의 작은 측정부터 해서 전제를 굳힌 뒤 결정할 것.
