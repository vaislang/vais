# 새 언어(Aria) 모호성 0 감사 — v0.1 명세

**작성일**: 2026-06-06
**대상**: `new-language-grammar-draft-2026-06-06.md` v0.1
**목적**: 명세가 진짜 "한 의미 = 한 표현"인지 점검. 모호성은 AI가 익숙한(다른 언어) 방식을 시도해
틀리게 만드는 근본 원인 (Vais 실패의 핵심).
**판정 기준**: 같은 의미를 **두 가지 이상으로 쓸 수 있으면** 모호성(AMBIGUOUS). AI가 어느 쪽을 쓸지
예측 불가 → cold-start 위험.

> 정직성 고지: 이 명세는 내가 만들었다. 감사에서 내가 "편의/직관 수용"으로 둔 다중-표현이
> 실제 모호성임을 인정하고 짚는다.

## 1. 발견된 모호성 (수정 필요)

### A-1. List 생성: `List<Int>::new()` vs `[10, 20, 30]` 🔴 AMBIGUOUS
- 명세가 **둘 다** 허용 (§2.5, line 103/111-112).
- 빈 리스트 만들 때: `List<Int>::new()` 인가 `[]` 인가? 둘 다 가능 → AI가 헷갈림.
- **이게 정확히 Vais 함정 4의 재발 위험**: turbofish 직관을 수용하려다 두 길을 열었다.
- **수정안**: **리터럴 `[...]`을 정식, 빈 것은 `[]: List<Int>`로 통일.** `::new()`는 제거하거나,
  반대로 `::new()`만 두고 리터럴 제거. **한 가지만.** (헌법 위반: 한 작업=한 문법 P3)
- 권고: **리터럴 우선** (`[1,2,3]`, 빈 건 `List<Int>::new()` 대신 `[]`로) — AI가 가장 자연스럽게 씀.

### A-2. match arm 본문: 표현식 vs `return` 🟡 SEMI-AMBIGUOUS
- 명세: arm은 표현식이지만 `=> return 1`도 허용 (§2.3 line 87).
- `Color.Red => 1` 과 `Color.Red => return 1` 둘 다 가능 → 어느 쪽?
- 단, 이건 **의미가 다름**: 전자는 match가 값을 내고, 후자는 함수를 즉시 반환. **모호성 아님**(다른 의미).
- 그러나 **혼동 위험**: AI가 "그냥 값 내려는데" `return`을 붙이거나 그 반대.
- **판정: 허용하되 명확히** — 둘은 의미가 다르므로 둘 다 필요. 단 에러/문서에서 구분 강조.
  (P6 "흔한 패턴 ceremony 없이"와 충돌 안 함.)

### A-3. 논리 연산: 단어 `and/or/not` 만? 🟢 OK (단 확인)
- 명세는 `and or not` 단어만, `&& || !` 금지 (line 30).
- **모호성 없음** — 한 가지. 단 AI가 Rust 직관으로 `&&`를 쓸 위험 → 에러가 `help: use 'and'` 줘야 함.
- **판정: OK.** (P4 에러 안내로 보완)

### A-4. 가변 바인딩: `let mut` 만? 🟢 OK
- `let`/`let mut` 한 가지 (line 32, 49). Vais의 `:=`+`~mut` 모호성 제거됨.
- **판정: OK — 모호성 0 달성.**

### A-5. 타입 변환: `Int(x)` 만? 🟢 OK
- 명세: 명시적 `Int(x)`/`F32(x)`만 (§2.8, §3). `as` 같은 대안 없음.
- **판정: OK** — 단 AI가 Rust `x as Int` 시도 위험 → `help: use 'Int(x)'`.

### A-6. enum 접근: `Color.Red` 만? 🟢 OK (강점)
- `.` 한 가지로 통일 (line 86). `::`도 unqualified도 금지.
- **판정: OK — Vais 실패 3을 정확히 차단.** 단 Rust `::` 시도 위험 → `help: use '.'`.

### A-7. 함수 호출 vs 메서드: `v.sum()` vs `sum(v)`? 🟡 미정의
- 명세가 `v.sum()` 메서드만 보여주나, 자유함수 `sum(v)`도 가능한지 불명확.
- **모호성 위험**: AI가 둘 다 시도 가능.
- **수정안**: **메서드 체인을 정식**으로 (`v.sum()`, `v.map(..).filter(..)`), 자유함수 형태는 안 만든다.
  명세에 "컬렉션 연산은 항상 메서드"라고 명문화 필요.

### A-8. 반복: `for`/`while`/`loop` 셋 다 🟡 의미 분리 필요
- 명세에 셋 다 있음 (line 65-66 for; reference에 while/loop).
- **모호성 아님**(의미 다름: for=범위, while=조건, loop=무한) — 단 **경계 명확화 필요**:
  "정해진 범위→for, 조건→while, 무한+break→loop. 한 상황엔 한 가지." 명문화.

## 2. 누락(명세 미정의) — 모호성의 잠재 원천

명세 v0.1이 **정의 안 한** 것들 (AI가 추측 → 제각각 = 모호):
- 문자열 보간 (`"{x}"` 인가 `format(...)`인가?) — Vais는 보간, Rust는 `format!`. **정해야 함.**
- 에러 처리 (`?` 연산자? `Result` match만?) — Vais/Rust 직관 충돌 가능. **정해야 함.**
- 가시성 (`pub` 위치/세분성) — keyword만 있고 규칙 미정.
- Option/Result 표준형 (T5 과제가 Option을 우회한 이유).
- 제네릭 제약 (`fn f<T: Trait>` 문법).
- 모듈/import (P5 prelude 외 사용자 모듈은?).

→ **이것들이 v0.2에서 "한 가지로" 정의돼야 한다.** 미정의는 곧 모호성이다.

**핵심 결론**:

- **다행히 대부분은 "내가 편의로 둘 다 허용한 것"** — 한 가지로 정하면 해결된다. 구조적 결함 아님.
- **교훈(메타)**: "Rust 직관 수용"(A-1 turbofish, A-7)이 모호성을 만든다. **직관 수용 ≠ 두 길 허용.**
  직관에 맞는 **한 가지**를 고르는 것이 옳다. 둘 다 열면 Vais 실패를 반복한다.

## 4. v0.2 수정 지침 (이 감사의 산출)
1. **A-1**: List 생성 한 가지로 (리터럴 `[...]` 정식 + 빈 건 명확한 한 형태).
2. **A-7**: 컬렉션 연산은 메서드 체인만 (자유함수 금지) 명문화.
3. **A-8**: for/while/loop 용도 경계 명문화.
4. **누락 6종**: 문자열보간/에러처리/가시성/Option·Result/제네릭제약/모듈을 각각 **한 가지**로 정의.
5. **공통**: 모든 "Rust 직관 위험" 지점에 `help:` 에러 (`&&`→`and`, `::`→`.`, `as`→`Int(x)`).
6. 원칙: **"둘 다 허용"이 보이면 즉시 하나로.** 편의보다 모호성 0.
