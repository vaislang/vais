# Vais AI-Native 경쟁력 종합 평가 보고서

**작성일**: 2026-06-06
**근거**: 본 보고서의 모든 정량 수치는 이 세션에서 `vaisc` / `rustc`로 **실측**한 값이다.
추측·시장예측은 그 자체로 명시한다.

---

## 0. 한 줄 결론

> **"Vais를 통해 명확한 문법 예제 케이스를 충분히 만들어 두면 AI가 잘 쓸 수 있다"는 가설은
> 이 세션의 실측으로 (조건부) 지지된다.** 단, 그것이 곧 "범용 Rust 대체 경쟁력"을 의미하지는
> 않는다. 두 명제는 분리해서 봐야 한다.

---

## 1. 실험 데이터 (실측)

### 1.1 실험 A — Cold-start 생성 정확도 (5 과제)
세션 초반, Vais를 "방금 접한 AI"처럼 첫 시도 생성.

| 과제 | VAIS | RUST |
|------|------|------|
| 짝수 합 | ✓ 1st | ✓ 1st |
| struct 메서드 | ✓ 1st | ✓ 1st |
| 문자열 카운트 | ✓ 1st | ✓ 1st |
| 제네릭 max | ✓ 1st | ✓ 1st |
| **Vec 합계** | **✗ 3회 실패** (turbofish + import 누락 + std-path) | ✓ 1st |

- **Cold-start: VAIS 4/5, RUST 5/5.**
- 코드 크기: 단순 과제는 Vais가 약간 작으나, stdlib 과제(문자열/Vec)는 Rust가 압도(iterator). **총합 Rust 22% 짧음.**

### 1.2 실험 B — 학습 후 생성 정확도 (10 과제)
세션 후반, AI(나)가 이미 Vais 함정을 학습한 상태.

| 지표 | VAIS | RUST |
|------|------|------|
| 첫시도 컴파일 | **10/10** | 9/10 |
| 런타임 정답 | **10/10** | 9/10 (r10 f64 리터럴 타입 실수로 컴파일 실패) |

- **학습 후: VAIS가 RUST를 앞섬.**
- **이 반전이 핵심 데이터다.**

### 1.3 실험 C — 컴파일 속도 (함수 N개, 캐시 제거, 충돌 없는 이름)

| 규모 | rustc | vaisc(LLVM) | 비고 |
|------|-------|-------------|------|
| 5 함수 | 129ms | 128ms | 동급 |
| 200 함수 | 175ms | 151ms | 동급(미세 우위) |
| 500 함수 | 233ms | 208ms | 동급 |
| 1000 함수 | 342ms | 368ms | 동급 |
| **check만 (codegen 제외)** | **64ms** | **33ms** | **Vais 2배 빠름** |

- 전체 빌드(LLVM 포함)는 **동급**. 둘 다 LLVM이 시간의 절반 이상.
- **프론트엔드(렉서+파서+타입체크)는 Vais가 2배 빠름** (borrow checker 부재).

### 1.4 실험 D — AI Self-Correction (에러 메시지 품질)

| 버그 종류 | VAIS | RUST |
|----------|------|------|
| 타입 불일치 / 미정의 변수 / 인자 개수 | ✓ 동급 (Vais가 더 간결하기도) | ✓ |
| **turbofish / 메서드 오타 등** | ✗ `found X, expected Y` (증상만) | ✓✓ `help:` + **정확한 수정 코드** |
| Vais 고유 함정(단일문자 키워드 충돌 등) | 🔶 단서만 | (해당 없음) |

- 공통 에러는 Vais도 우수. **단 Rust의 `help:`형 자동수정 제안(수정 코드 제시)이 AI 회복의 gold standard이고 Vais엔 없다.**

## 2. 데이터가 말하는 것 (해석)

### 2.1 사용자 가설별 판정 (업데이트)

| 가설 | 초기 내 판단 | 실측 후 판정 |
|------|------------|------------|
| "명확한 예제/문서를 만들면 AI가 잘 쓴다" | 회의적 | **🔶 조건부 참** — 학습 컨텍스트 충분하면 AI가 10/10. 세션 중 내가 직접 증명(처음 틀림→학습 후 완벽) |
| "토큰 수를 줄인다" | 회의적 | **❌ 반증** — 단일문자 절약 < stdlib 부재 페널티. Rust가 총 22% 짧음 |
| "단일문자 = AI에 치명적" | 강하게 주장 | **🔶 과했음** — cold-start만 어렵고, 학습 후엔 불리하지 않음 |
| "성능 압도" | 불가 | **❌ 사실상 불가** — 같은 LLVM, Rust가 이미 정점 |
| "컴파일 속도 압도" | 가능성 | **🔶 잠재력 있음** — 현재 동급이나 프론트엔드 2배 빠름 → LLVM 우회 시 압도 가능 |

### 2.2 핵심 통찰

1. **"AI가 신규 언어를 못 쓴다"는 내 초기 주장은 데이터로 약화됐다.**
   학습 컨텍스트(좋은 예제+문서)가 충분하면 AI는 정확히 쓴다. 사용자 직관이 옳은 부분이 크다.

2. **단, "잘 쓴다"의 전제는 "충분한 예제가 컨텍스트에 있다"이다.**
   - 이건 **달성 가능**하다: LIVING_SPEC 같은 검증된 예제 코퍼스 + AI가 생성→검증→정답쌍 자동 축적.
   - 즉 "예제 케이스를 Vais로 명확히 만들어 둔다"는 전략이 **데이터상 유효**하다.

3. **그러나 "AI가 잘 쓴다" ≠ "Rust를 대체한다".** 이 둘을 혼동하면 안 된다.

## 3. "범용 Rust 대체" 경쟁력 — 정직한 평가

### 3.1 코드로 메울 수 있는 격차 (AI로 가속 가능)
- 컴파일러 기능 (현재 selfhost는 certified core만 / Rust vaisc 17.4만 LOC)
- 표준 라이브러리 (현재 82 파일 / Rust std 방대)
- 에러 메시지 품질 (`help:` 자동수정 추가 — Vais도 공통 에러는 이미 우수)
- 컴파일 속도 (LLVM 우회 백엔드)

→ **이것들은 "코드 작성 문제"라 AI(나)로 가속된다.** 이 세션에 버그 8개를 빠르게 고친 게 증거.

### 3.2 코드로 메울 수 없는 격차 (AI로 불가 — 가장 큰 해자)
- **생태계**: crates.io 15만+ 패키지 vs 사실상 0. 이건 코드량이 아니라 **사람들의 집단 노동+시간+신뢰**. AI가 15만 개를 짜도 "유지보수·커뮤니티"가 없다.
- **채택/신뢰**: 기업이 "1인+AI가 짠 언어"를 프로덕션에 쓰는 결정. 사회적 문제.
- **학습데이터의 깊이**: Rust는 문법+수백만 예제가 LLM 가중치에 박혀 있어 cold-start가 강하다. Vais는 컨텍스트 주입으로 보완하나 동등하진 않다.

### 3.3 역사적 참조 (추측 아닌 사실)
- 성공한 신규 시스템 언어(Rust/Go/Swift)는 **전부 거대 조직 후원**(Mozilla/Google/Apple)이 있었다.
- 1인/소규모로 "범용 Rust 대체"에 성공한 사례는 사실상 없다.
- Zig조차 수년+재단+다수 기여자로도 "Rust 대체"가 아니라 "C 대체+니치" 포지션.

### 3.4 결론
> **"범용 Rust 대체"라는 시장 목표로서는 — 정직히 성공 확률이 매우 낮다.**
> Vais의 기술 품질 문제가 아니라, 거의 모든 신규 시스템 언어가 부딪히는 구조적 벽(생태계 해자)이다.
> 단, **기술적 따라잡기(컴파일러 동등) 자체는 AI로 가능**하다 — 그게 "대체"로 이어지지 않을 뿐.

## 4. 진짜 기회가 있는 포지션 (재프레임)

데이터가 가리키는, "범용 대체"보다 현실적인 차별화 축:

### 4.1 "AI 폐쇄 루프 검증가능 언어"
- AI가 코드를 쓰고 → 언어가 즉시 검증하고 → AI가 피드백으로 고치는 루프가
  인간 언어보다 빠르고 정확한 언어.
- 여기선 **생태계 해자가 약하다** (AI가 일회용 코드를 대량 생성·실행하는 환경).
- 승부처: **컴파일 속도(루프 속도) + 자동수정 에러 + 모호성 0.**

### 4.2 차별점은 "토큰"이 아니라 다음 3가지
실측이 가리키는, AI가 잘 쓰는 언어의 진짜 조건:
1. **모호성 0 / 한 가지 쓰는 법** — AI는 선택지·함정이 많으면 틀린다.
   (Vais의 단일문자 키워드 충돌 `A`=async, `~`=mut, `f32`=타입 등이 정확히 이 함정.)
2. **`help:`형 자동수정 에러** — 에러가 AI에게 고치는 법을 그대로 준다 (Rust의 최강점, Vais는 미완).
3. **초고속 컴파일** — AI self-correction 루프 라운드당 시간이 핵심. LLVM 우회 디버그 백엔드.

### 4.3 "예제 코퍼스" 전략 (사용자 가설의 핵심)
- 실측상 **유효**: 충분한 검증된 예제가 컨텍스트에 있으면 AI가 정확히 쓴다.
- 구현: LIVING_SPEC 확장 + AI 생성→`vaisc build` 검증→정답쌍 자동 축적 → cold-start 약점 보완.
- **이것이 "학습데이터 부재"를 부분적으로 푸는 길이다.**

## 5. 권고

### 5.1 성공 기준을 분리하라
| 프레임 | 성공 가능성 | 비고 |
|--------|-----------|------|
| 학습/연구/포트폴리오 | ✅ 이미 성공 | self-host+DB+서버+웹 = 극소수만 해낸 일. 경쟁력과 무관하게 최고 가치 |
| AI-native 니치 (폐쇄 루프) | 🔶 가능성 있음 | 단 핵심 가설 추가 검증 + 선점 경쟁(Mojo 등) |
| 범용 Rust 대체 | ❌ 매우 낮음 | 생태계 해자. 1인+AI로 불가에 가까움 |

### 5.2 만약 계속 간다면 — 방향
- **표면 설계 재고**: 단일문자 키워드는 cold-start와 모호성에서 손해. "모호성 0 + 영어스러움"이 AI 정확도에 유리(데이터). 토큰 절약 베팅은 폐기 권장.
- **백엔드는 재활용**: Vais의 타입체커/codegen/self-host 인프라는 자산. 빠른 디버그 백엔드(LLVM 우회) 추가가 최우선 기술 레버.
- **에러에 `help:` 자동수정 추가**: AI self-correction의 결정타. Vais 고유 함정마다 수정 제안.
- **예제 코퍼스 자동 축적**: 사용자 가설의 핵심. AI 생성→검증→정답쌍 파이프라인.

### 5.3 다음 결정적 측정 (아직 미측정)
- **cold-start self-correction 라운드 수**: 신규 AI가 에러를 받아 몇 라운드에 수렴하는가 (Vais vs Rust).
- **예제 코퍼스 효과 정량화**: 예제 N개를 컨텍스트에 줄 때 cold-start 정확도가 얼마나 오르는가.
- 단, 시장·채택·생태계는 코드로 측정 불가 — 이 변수들이 "범용 대체"의 진짜 결정자임을 인지.

## 6. 가장 정직한 마무리

- 사용자의 직관("예제를 명확히 만들면 AI가 쓸 수 있다") — **실측상 맞다.**
- 내 초기 회의("AI가 신규 언어를 못 쓴다") — **과했고, 데이터로 약화됐다.**
- 그러나 **"AI가 Vais를 잘 쓴다"와 "Vais가 Rust를 대체한다"는 다른 명제**다.
  전자는 달성 가능, 후자는 생태계 해자로 매우 어렵다.
- **이 작업의 가치는 "Rust 대체 성공 여부"와 무관하게 크다** — 그 둘을 묶지 말 것.

> 데이터는 "불가능"도 "확실한 성공"도 말하지 않는다. "AI 활용 + 예제 코퍼스로 기술적 따라잡기는
> 가능하나, 범용 시장 대체는 생태계 때문에 별개의 (훨씬 어려운) 문제"라는 **혼합 결론**을 말한다.
