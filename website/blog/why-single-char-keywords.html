The Design Philosophy Behind Single-Character Keywords

One of the most striking features of Vais is its single-character keyword system. Instead of function, we write F. Instead of struct, we write S. This is not merely an aesthetic choice -- it is a deliberate design decision rooted in the intersection of AI code generation, information theory, and developer ergonomics.

The Token Economy

Large Language Models process text in tokens. Every token costs compute, latency, and money. When an LLM generates code, the keyword function consumes one token (sometimes two, depending on the tokenizer). In Vais, F is a single character that consistently maps to one token.

This matters at scale. Consider a typical 50-function program:

LanguageKeyword TokensSavings
Rust~350 keyword tokensbaseline
Go~280 keyword tokens~20%
Vais~120 keyword tokens~65%

The reduction comes not just from shorter keywords, but from Vais's expression-oriented design that eliminates many keywords entirely. return is often implicit (last expression is the return value), and type annotations are inferred.

Side-by-Side: Vais vs. Rust

Consider a generic function with pattern matching:

Rust (72 tokens):

fn classify(x: i64) -> i64 {
    match x {
        0 => 100,
        1..=9 => 200,
        _ => 300,
    }
}

Vais (48 tokens):

F classify(x: i64) -> i64 {
    M x {
        0 => 100,
        1..9 => 200,
        _ => 300
    }
}

The Vais version is 33% fewer tokens while maintaining identical readability for developers who know the keyword map. The learning curve is approximately 15 minutes -- memorizing 13 single-letter mnemonics.

The @ Self-Recursion Operator

The @ operator is perhaps our most distinctive design choice. It calls the current function recursively without naming it: 

# Fibonacci in one line
F fib(n:i64)->i64 = n<2 ? n : @(n-1) + @(n-2)

This eliminates the common pattern where a function name appears in both the definition and the recursive call. The @ symbol is universally recognizable and unambiguous in its meaning.

Expression-Oriented by Default

In Vais, everything is an expression. The last expression in a block is its return value, eliminating explicit return in most cases: 

# No explicit return needed
F abs(x: i64) -> i64 = I x < 0 { -x } E { x }

# Ternary for simple cases
F max(a: i64, b: i64) -> i64 = a > b ? a : b

Design Trade-offs We Considered

This approach is not without trade-offs. We carefully evaluated each concern:

  • Readability for newcomers: The initial "alphabet soup" impression fades within one coding session. LSP hover information shows the full keyword name.
  • Collision with variable names: All Vais keywords are uppercase single letters. Variable names conventionally start with lowercase, creating a natural visual separation.
  • IDE support: The VSCode extension and IntelliJ plugin provide syntax highlighting that makes keywords visually prominent.
  • Searchability: F (with trailing space) is more searchable than you might expect. LSP "Go to Definition" handles the rest.

The AI-Native Perspective

As AI models become primary authors of code, the cost function shifts. Traditional readability optimized for human scanning speed. AI-native readability optimizes for:

  1. Token efficiency -- fewer tokens mean faster generation and lower cost
  2. Unambiguous parsing -- single-character keywords have zero ambiguity
  3. Information density -- more logic per context window means better AI reasoning

Vais is designed for the era where an AI writes the code and a human reviews it. The single-character keywords make generation efficient, while the LSP tooling makes review ergonomic.
