The Self-Hosting Journey: 50,000 Lines of Vais Compiling Itself

Self-hosting -- a compiler written in the language it compiles -- is the traditional rite of passage for a programming language. It proves that the language is expressive enough to build complex software, and that the compiler is correct enough to compile non-trivial programs. This is the story of how Vais achieved self-hosting with 50,000+ lines of code and 100% clang compilation success.

Why Self-Host?

A self-hosting compiler serves three purposes:

  1. Proof of expressiveness: If you can build a compiler in a language, you can build anything. A compiler requires data structures, string processing, pattern matching, error handling, file I/O, and code generation -- it exercises every feature.
  2. Proof of correctness: A self-hosting compiler is the ultimate integration test. If the generated LLVM IR compiles and the resulting binary produces correct output, the compiler is working.
  3. Dogfooding: Using Vais to build its own compiler revealed dozens of ergonomic issues, codegen bugs, and missing features that would have been invisible from the outside.

The Bootstrap Architecture

The Vais self-hosting compiler lives in the selfhost/ directory with 50,000+ lines across multiple modules: 

selfhost/
  main.vais          # Entry point
  lexer.vais         # Token scanner (114 tests)
  parser.vais        # Recursive descent parser
  ast.vais           # AST node definitions
  types.vais         # Type checker
  codegen.vais       # LLVM IR emitter
  ir_builder.vais    # IR construction helpers
  builtins.vais      # Built-in function definitions
  ...

The bootstrap compiler reads .vais source files, lexes them, parses an AST, runs type checking, and emits LLVM IR that is compiled by clang into a native binary.

The Milestones

The self-hosting journey progressed through phases, each adding capability:

Phase 1-10 Basic lexer, parser, and arithmetic codegen. The compiler could tokenize and emit IR for simple expressions.
Phase 11-20 Control flow (if/else, loops, match), functions, and struct definitions. The compiler could compile meaningful programs.
Phase 21-30 Generic types, trait dispatch, error handling (Result/Option), and string processing. The type system became self-sufficient.
Phase 31-40 Full bootstrap achieved. The compiler could compile itself, producing valid LLVM IR. 21/21 clang compilation success. Module system, imports, and standard library integration.
Phase 41-56 Optimization and hardening. MIR lowering, advanced optimization passes, extensive E2E testing (1,200+ tests), and codegen correctness improvements.

The Hardest Bugs

Self-hosting exposed bugs that no test suite would have caught:

1. The "Everything is i64" Problem

Early Vais codegen treated almost everything as i64. This worked for simple programs but created subtle bugs in the self-hosting compiler where enum tags, boolean values, and pointers all needed distinct LLVM types. Phase 42 systematically addressed 135+ instances of this pattern.

2. Void Phi Nodes

LLVM's phi instruction cannot use the void type. When the self-hosting compiler generated code for blocks ending in void-returning functions, it produced invalid IR: phi void. The fix was to emit add i64 0, 0 as a placeholder in void contexts.

3. Generic Closing Brackets

The expression Vec<HashMap<K,V>> has two closing angle brackets >>, which the lexer tokenized as a single right-shift operator. The fix introduced a pending_gt flag that splits >> into two > tokens when inside a generic context.

4. Occurs Check

The type inference engine lacked an occurs check, causing infinite recursion during unification when a type variable appeared in its own constraint. This is a classic bug in Hindley-Milner implementations, and it only surfaced when the self-hosting compiler tried to infer types for its own recursive data structures.

Lessons Learned

  • Write the self-hosting compiler early. We started at Phase 1. Every new language feature was immediately tested by implementing something in the self-hosting compiler. This provided continuous feedback.
  • Test at the IR level. Many bugs only manifest when the generated LLVM IR is validated by clang. Unit tests that only check AST or type information miss codegen issues.
  • The compiler is the best benchmark. Compiling 50K lines revealed performance bottlenecks that synthetic benchmarks never would. The hot-path optimization work (Phase 26) was entirely motivated by self-hosting profiling.
  • Bootstrap in increments. Each phase added one capability and verified it worked. We never tried to "big bang" the bootstrap.

What Self-Hosting Proves

Self-hosting proves that Vais is not a toy language. It has the expressiveness to build systems software (a compiler is among the most demanding applications), the correctness to produce valid machine code, and the performance to do it in under 65 milliseconds.

The selfhost/ directory is open source. You can read the self-hosting compiler, modify it, and bootstrap it yourself. It is both the ultimate test and the best documentation for what Vais can do.
